Following the previous day, I realised that the dependency I used for the error messages was using the default implementation - not mine - which meant it returned the default status code 500 with no body. To fix this, I create a pull request to add support for this.


a screenshot of the expected body, note: the high latency is due to being in debug mode

After all of the fixes, the user API now functions with no errors. The /me endpoint returns the expected result, the /logout endpoint successfully logs out the user (and removes the cookie).

Now, we have to test the validation and for any possible SQL injections when it comes to logging in and registration.

In the future, I plan to also add CAPTCHA when it comes to these critical endpoints to prevent abuse and implement rate limits too. I could also add an extractor for the payloads to automatically validate the inputs.

  • Implement FromRequest for request payloads

While testing I realised if a user logs in twice, the previous session will be invalidated. Therefore a user can only use the application on one device at a time, having to re-login when switching devices. To fix this, I need to store many invalidation tokens, within an array.

  • Use an array for the invalidation tokens
